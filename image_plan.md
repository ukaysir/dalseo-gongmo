## 1. 제작 목표

제안서에는 설명용 이미지 3장만 사용한다. 각 이미지는 서로 다른 질문에 답해야 한다.

1. **그림 1:** 현장사진이 어떤 행정 흐름을 거쳐 결과검증까지 이어지는가?
2. **그림 2:** 조사자가 AI 도움을 받아 어떻게 정확하게 촬영·재촬영하는가?
3. **그림 3:** 담당자가 관리주체·처리경로·조치결과를 어떻게 사건 단위로 관리하는가?

세 이미지는 앱 기능을 많이 보여 주는 화면이 아니라 다음 핵심 논리를 시각화한다.

> 현장수집 → AI 품질·사건 정리 → 규칙 기반 관리주체 분기 → 사람 승인 → 행정조치 → 동일지점 재촬영·전후검증

서비스의 상위 개념은 `행정조치`로 통일한다. 물리적 수선이 필요한 경우에만 `보수`라고 쓰며, 확정 슬로건인 `수집에서 끝나는 데이터를, 보수까지 이어지는 행정으로`는 유지한다.

```text
   ├─ 길눈AI_01_전체운영흐름.html
   ├─ 길눈AI_01_전체운영흐름.png
   ├─ 길눈AI_02_현장촬영_AI검수.html
   ├─ 길눈AI_02_현장촬영_AI검수.png
   ├─ 길눈AI_03_행정작업카드_전후검증.html
   └─ 길눈AI_03_행정작업카드_전후검증.png
```

- HTML은 외부 URL 없이 열려야 한다.
- 아이콘·예시 지도·탐지 프레임은 인라인 SVG로 제작하고, 보도·점자블록 현장과 전후사진은 이미지 생성 후 HTML/CSS 안에 배치한다.
- PNG는 투명 배경이 아닌 흰색 배경으로 저장한다.
- 기존 그림 1 HTML·PNG는 새 운영구조에 맞춰 개정한다.

### 3-2. 서체와 크기

- 서체: 로컬 `Pretendard Variable` 고정
- 메인 제목: 52~56px / 800 이상
- 부제: 25~28px / 500~600
- 카드 제목: 30~34px / 700~800
- 주요 UI 문구: 24~28px / 600~700
- 설명 문구: 21~24px / 500~600
- 출처·가상 데이터 표시: 18~20px / 500 이상
- 본문에 삽입했을 때 읽히도록 주요 정보는 24px 미만으로 만들지 않는다.

### 3-3. 레퍼런스 반영 규칙

- 흰색 캔버스, 네이비 제목, 청록·파랑·보라의 제한된 강조색을 사용한다.
- 카드 바깥은 흰색으로 두고 옅은 테두리와 약한 그림자만 사용한다.
- 카드 상단 컬러 바, 영문 메타 문구, 장식용 점은 사용하지 않는다.
- 과도한 그라데이션과 입체 그림자는 사용하지 않는다.
- 카드 하단 주석은 배경 없는 `* 문구` 형식으로 우측 정렬한다.
- 자료 출처는 전체 이미지의 우측 하단에 `* 자료:` 형식으로 표시한다.
- 하단 네이비 핵심 메시지 띠는 내용 전달에 필요할 때만 사용한다.

## 5. 그림 1 — 전체 운영 흐름도

### 5-1. 목적

기존의 단순 선형 흐름을 개정해 `관리주체 확인` 뒤 처리경로가 갈라지고, 모든 경로가 결과확인으로 다시 모이는 구조를 보여 준다.

### 5-2. 화면 구조

1. **현장의 눈**
   - 2026년 사업자료상 배치 인원 합계 `177명`
   - 현장사진 촬영

2. **AI 보조**
   - 사진 품질검수
   - 장애유형 후보
   - 동일지점 사건 통합
   - 정해진 항목 기반 조치 우선순위

3. **관리주체 확인 — 사람 승인 지점**
   - 규칙·공간정보로 관리주체와 처리경로 후보 제시
   - 담당자가 최종 확인

4. **처리경로 분기**
   - 직접 조치
   - 현장 확인·실측
   - 타기관 이첩
   - 장기검토·조치곤란

5. **결과검증**
   - 동일지점 재촬영
   - 전후사진 확인
   - 완료 또는 재조치

### 5-3. 도식 형태

```text
현장 촬영
   ↓
AI 품질검수·사건 통합
   ↓
관리주체 확인 ◇
   ├─ 직접 조치 ─────────┐
   ├─ 현장 확인·실측 ───┤
   ├─ 타기관 이첩 ──────┤ → 재촬영·전후검증
   └─ 장기검토 ─────────┘
```

- 분기점은 마름모형 SVG로 표현한다.
- 생성형 AI가 관리주체를 결정하는 것처럼 표현하지 않는다.
- 분기 영역에는 `규칙 기반 후보 · 담당자 확정`을 표시한다.
- 하단 핵심 문구는 확정 슬로건을 사용한다.

### 5-4. 사용할 데이터와 출처

| 정보 | 구분 | 사용 방식 |
|---|---|---|
| 177명 | 검증된 실제 수치 | `2026년 사업자료상 배치 인원 합계`라고 표시 |
| 4개 수행기관 | 검증된 실제 정보 | 필요하면 출처 주석에만 표시 |
| 처리경로 | 제안 기능 | 직접 조치·실측·이첩·장기검토 분기 |
| AI·행정 단계 | 제안 기능 | 숫자 성과 없이 기능명만 표시 |

우측 하단 출처:

## 6. 그림 2 — 조사자 현장촬영·AI 검수 화면

### 6-1. 목적

고령 조사자를 고려한 큰 글씨·음성 중심 UI와, 전후검증을 위한 같은 구도 재촬영 보조를 보여 준다. 실제 작동 앱이 아니라 세 개의 정적 모바일 화면을 한 장에 배치한다.

### 6-2. 화면 구성

#### 화면 A — 오늘의 조사

- `오늘의 조사구역`과 `조사순서`
- 기수집 구간과 조사 후보구간을 구분한 도식형 지도
- 실제 안전경로 안내는 넣지 않는다.
- 실제 지도 API를 사용하지 않고 도로선·시설 아이콘을 SVG로 그린다.

#### 화면 B — 촬영·재촬영 가이드

- 큰 촬영 버튼
- 점자블록 파손 현장을 크게 보여 주는 생성 이미지 카메라 미리보기
- 이전 사진의 반투명 윤곽 오버레이
- 촬영방향 화살표와 대상 위치 가이드
- GPS 오차 경고
- `한 걸음 가까이 촬영해 주세요` 같은 큰 글씨 안내
- 얼굴·차량번호 흐림 처리 아이콘

#### 화면 C — AI 후보 확인

- 세 유형 중 하나의 후보: 보도 파손·점자블록 파손·가림·통행 방해 장애물
- `AI 분석 예시 · 담당자 확인 전`
- `맞아요`, `수정`, `재촬영`에 해당하는 정적 버튼
- 법정 단차나 경사 수치는 표시하지 않는다.

### 6-3. 사용할 데이터

이 그림에는 실제 행정사건이나 실제 위치를 사용하지 않는다.

| 항목 | 표시 예시 | 데이터 구분 |
|---|---|---|
| 위치 | `달서구 예시지점 A` | 가상 위치 |
| 사건번호 | `GN-EX-001` | 가상 데이터 |
| 장애유형 | `점자블록 가림 후보` | AI 분석 예시 |
| 촬영안내 | `점자블록 전체가 보이게 촬영해 주세요` | 제안 기능 |
| 상태 | `재촬영 필요`, `담당자 확인 전` | 화면 예시 |

화면 상단 또는 우측 하단에 다음 문구를 고정한다.

> 가상 데이터 기반 화면 예시 · 실제 구현 화면 아님

## 7. 그림 3 — 행정 작업카드·전후검증 통합 화면

### 7-1. 목적

길눈AI가 단순 탐지·지도 서비스가 아니라 관리주체 배정, 처리경로, 이력과 결과검증을 관리하는 행정 운영모델임을 보여 준다.

### 7-2. 레이아웃

- 상단 25%: 사건 요약과 상태
- 좌측 35%: 작업카드·규칙 기반 관리주체 분기
- 우측 65%: 조치 전후사진과 변화영역
- 하단: 접수부터 재확인까지 처리이력 타임라인

### 7-3. 필수 요소

1. **사건 요약**
   - 사건번호, 가상 위치, 유형 후보
   - 동일지점 사진 묶음 표시
   - `가상 데이터`, `AI 분석 예시`, `담당자 확인 전`

2. **조치 우선순위 근거**
   - 장벽 위험도
   - 장애인 관련시설·노인복지시설·정류장 인접성
   - 반복성·지속성
   - 숫자 총점보다 항목별 근거를 우선 표시

3. **관리주체·처리경로**
   - 입력 근거: 위치, 시설유형, 확인된 관리대장·업무규칙
   - 출력 후보: 구청 직접 조치, 현장 실측, 타기관 이첩, 장기검토
   - `규칙 기반 후보`와 `담당자 확정`을 분리 표시
   - 생성형 AI는 `확정 정보로 문장 초안 생성` 역할만 표시

4. **조치 전후사진**
   - 같은 SVG 보도 장면의 조치 전·후 버전
   - 전: 파손·가림 영역에 점선 또는 색상 표시
   - 후: 장애가 제거되거나 수선된 상태
   - 이미지 정합·변화탐지 후보영역 표시
   - 담당자 선택: `완료`, `재조치`, `타기관 처리 확인`

5. **처리이력 타임라인**
   - 접수 → 담당자 검토 → 관리주체 확정 → 배정·이첩 → 조치 → 재확인
   - 처리일과 소요기간은 모두 가상 데이터로 표시

### 7-4. 사용할 데이터

| 항목 | 사용 규칙 |
|---|---|
| 사건번호·위치·날짜 | 모두 가상 데이터 |
| 우선순위·반복 횟수·처리기간 | 가상 예시 표시 필수 |
| 시설·정류장 인접성 | 공개데이터 연계 제안임을 표시, 실제 연동처럼 표현하지 않음 |
| 관리주체·처리경로 | 규칙 기반 후보이며 담당자가 확정 |
| 조치 전후사진 | 직접 제작한 SVG 예시 장면 |
| 완료·재조치 상태 | 화면 이해를 위한 가상 상태 |

공공데이터 출처는 작은 주석으로만 표시한다.

- 달서구 장애인 관련시설 현황[8]
- 달서구 노인복지시설 현황[9]

- 접수·배정·조치 날짜
- 처리기간과 완료율
- 조치 전후 상태

### 8-3. 사용하지 않을 정보

- 실제 주민 얼굴·차량번호·개인 이동경로
- 라이선스가 확인되지 않은 지도·사진·아이콘
- 실제 달서구 부서가 사건을 승인한 것처럼 보이는 표현
- AI 정확도, 시간절감률, 비용절감률
- 일반 사진으로 계산한 법정 단차·경사·통행 폭
- 주민용 처리상태 공개 화면

---

## 9. SVG 제작 항목

공통으로 재사용할 인라인 SVG 요소를 먼저 만든다.

- 카메라, AI 칩, 사람 승인, 작업카드, 이첩, 재촬영 아이콘
- 보도 파손·점자블록·통행 장애물 예시 장면
- 조치 전·후 장면과 변화영역 마스크
- 도식형 지도와 시설·정류장 아이콘
- 관리주체 분기 마름모와 연결 화살표
- 처리이력 타임라인 아이콘
- 이전 사진 반투명 오버레이와 촬영방향 가이드

외부 SVG 라이브러리를 불러오지 않고 단순 선형 아이콘을 직접 작성한다.

---

## 10. HTML/CSS 제작 방식

1. 각 HTML에 고정 크기 `#artboard`를 둔다.
2. 공통 토큰은 `design-system.css`에서 관리한다.
3. Pretendard를 로컬 `@font-face`로 불러온다.
4. 모든 아이콘·장면·지도는 HTML 내부 SVG로 삽입한다.
5. 애니메이션, 클릭, 입력, API 요청은 구현하지 않는다.
6. 캡처 전 `document.fonts.ready`를 확인한다.
7. 캡처용 HTML은 스크롤과 외부 네트워크 의존성이 없어야 한다.

---

## 11. PNG 캡처 절차

1. Chrome 또는 Edge의 헤드리스 모드로 HTML을 연다.
2. 각 이미지의 정확한 `--window-size`를 지정한다.
3. `--force-device-scale-factor=1`, `--hide-scrollbars`를 적용한다.
4. 폰트와 SVG 렌더링이 끝난 뒤 PNG로 저장한다.
5. PNG 크기와 파일 용량을 확인한다.
6. 원본 해상도로 시각 검수한다.
7. 문서 삽입 폭 160mm로 축소한 상태에서도 주요 문구가 읽히는지 확인한다.
8. 검은 렌더링 영역, 잘린 글자, 폰트 대체가 있으면 재캡처한다.

## 14. 완료 기준

- HTML 3개와 PNG 3개가 `래퍼런스\generated`에 존재한다.
- 세 PNG의 픽셀 규격이 계획과 일치한다.

- 관리주체와 처리경로는 규칙 기반 후보이며 사람이 확정한다.

- 개인정보·법정치수·AI 정확도에 관한 과장 표현이 없다.
- 제안서 5분 발표에서 세 그림만으로 문제, 해결, 차별성과 효과를 설명할 수 있다.
